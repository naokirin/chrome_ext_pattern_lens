要素境界をまたぐ検索テストページ

このページの使い方:

Pattern Lens拡張機能の要素境界をまたぐ検索機能をテストするためのページです。

以下の各セクションで、HTMLの要素境界をまたいだテキストが正しく検索できることを確認してください。

1. インライン要素内の検索（要素境界なし）

テストケース1-1: span要素をまたぐテキスト（空白なし）

mkdir-p

検索文字列: mkdir-p

期待結果: 全体がハイライトされる（インライン要素間には自動スペース挿入なし）

テストケース1-2: 複数のspan要素をまたぐテキスト（明示的な空白あり）

Hello World!

検索文字列: Hello World!

期待結果: 全体がハイライトされる（DOMに空白が存在するため）

テストケース1-3: 異なるインライン要素の連続

npm install react

検索文字列: npm install react

期待結果: 全体がハイライトされる（要素間に空白あり）

2. ブロック要素をまたぐ検索

テストケース2-1: div要素をまたぐテキスト

Hello
World

検索文字列: Hello World

期待結果: 両方のdivの内容がハイライトされる（ブロック境界にスペースが挿入される）

テストケース2-2: p要素をまたぐテキスト

Lorem ipsum

dolor sit

検索文字列: ipsum dolor

期待結果: 両方のp要素にまたがってハイライトされる

テストケース2-3: 見出しと段落をまたぐテキスト

Important Note

This is critical

検索文字列: Note This

期待結果: 見出しと段落にまたがってハイライトされる

3. 複雑なネストをまたぐ検索

テストケース3-1: ネストしたspan要素

git commit -m "message"

検索文字列: git commit -m "message"

期待結果: 全体がハイライトされる

テストケース3-2: リスト項目をまたぐテキスト

  • First
  • Second
  • Third

検索文字列: First Second

期待結果: 最初の2つのリスト項目にまたがってハイライトされる

テストケース3-3: テーブルセルをまたぐテキスト

Cell One Two

検索文字列: Cell One

期待結果: 最初の2つのセルにまたがってハイライトされる

4. 正規表現を使った検索

テストケース4-1: 要素をまたぐ正規表現マッチ

test123

検索文字列（正規表現ON）: test\s*\d+

期待結果: 全体がハイライトされる

テストケース4-2: 複数要素にまたがる柔軟なパターン

email: user@example.com

検索文字列（正規表現ON）: email:\s*\w+@\w+\.\w+

期待結果: 全体がハイライトされる

5. Chrome標準検索との互換性テスト

テストケース5-1: 標準検索との比較

The quick brown fox

検索文字列: quick brown

期待結果: Pattern LensもChrome標準検索（Ctrl+F）も同様にマッチする

テストケース5-2: 複数のスペース

one two three

検索文字列: one two three

期待結果: スペースの数に関わらずマッチする

6. あいまい検索の正規化パターン（要素境界をまたぐ）

テストケース6-1: 全角・半角アルファベットの正規化

ＡＢＣdef

検索文字列（あいまい検索ON）: abc def

期待結果: 全角アルファベットが半角に正規化されてマッチする

テストケース6-2: 大文字小文字の正規化

Hello WORLD

検索文字列（あいまい検索ON）: hello world

期待結果: 大文字小文字を区別せずにマッチする

テストケース6-3: ひらがな・カタカナの相互一致

テスト ページ

検索文字列（あいまい検索ON）: てすと ページ

期待結果: カタカナがひらがなに正規化されてマッチする

テストケース6-4: 記号の表記ゆれ吸収

2024ー01ー01

検索文字列（あいまい検索ON）: 2024-01-01

期待結果: 全角の伸ばし棒（ー）がハイフン（-）に正規化されてマッチする

テストケース6-5: 濁点・半濁点の結合文字正規化

か゛ は゜

検索文字列（あいまい検索ON）: が ぱ

期待結果: 別文字の濁点・半濁点が結合文字として正規化されてマッチする

テストケース6-6: 日付フォーマットの差異吸収

2024／01／01

検索文字列（あいまい検索ON）: 2024/01/01

期待結果: 全角スラッシュ（／）が半角スラッシュ（/）に正規化されてマッチする

テストケース6-7: アクセント付き文字の正規化（フランス語）

café français

検索文字列（あいまい検索ON）: cafe francais

期待結果: アクセント付き文字（é, ç）が英語アルファベットに正規化されてマッチする

テストケース6-8: アクセント付き文字の正規化（スペイン語）

español

検索文字列（あいまい検索ON）: espanol

期待結果: ñがnに正規化されてマッチする

テストケース6-9: 代用表記の対応（ドイツ語: ß → ss）

Straße

検索文字列（あいまい検索ON）: strasse

期待結果: ßがssに正規化されてマッチする

テストケース6-10: 代用表記の対応（ドイツ語: ä → ae）

Müller

検索文字列（あいまい検索ON）: mueller

期待結果: üがueに正規化されてマッチする

テストケース6-11: 代用表記の対応（ドイツ語: ö → oe）

Köln

検索文字列（あいまい検索ON）: koeln

期待結果: öがoeに正規化されてマッチする

テストケース6-12: 代用表記の対応（フランス語: œ → oe）

cœur

検索文字列（あいまい検索ON）: coeur

期待結果: œがoeに正規化されてマッチする

テストケース6-13: 複数の正規化パターンの組み合わせ

ＴＥＳＴー２０２４／０１／０１

検索文字列（あいまい検索ON）: test-2024/01/01

期待結果: 全角アルファベット、全角数字、全角記号がすべて正規化されてマッチする

テストケース6-14: アクセント付き文字と代用表記の組み合わせ

Straße Müller

検索文字列（あいまい検索ON）: strasse mueller

期待結果: ß→ss、ü→ueの両方が正規化されてマッチする

テストケース6-15: カンマ区切りの数字（要素境界をまたぐ）

検索文字列（あいまい検索ON）: 1234.56

期待結果: カンマが削除され、小数点のみが残ってマッチする

テストケース6-16: ヨーロッパ形式の小数点（要素境界をまたぐ）

テストケース6-17: 全角区切り文字を含む数字（要素境界をまたぐ）

１，２３４．５６

検索文字列（あいまい検索ON）: 1234.56

期待結果: 全角数字が半角に変換され、全角カンマが削除、全角ピリオドが半角ピリオドに変換されてマッチする

テストケース6-18: 数字と文字の混在（要素境界をまたぐ）
